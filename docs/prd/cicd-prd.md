# PRD: CI/CD 任务预热代理与 Runtime 二级调度系统

---

## 1. 背景与问题

### 1.1 当前状态
- 当前所有 CI 任务仍采用传统模型：**每次触发均为每个任务创建一个新 K8s Pod**。

### 1.2 核心痛点
虽然失败原因已能被快速诊断，但**执行过程本身的延迟**成为新的瓶颈：
- **Pod 冷启动时间过长**：调度、镜像拉取、容器初始化通常耗时 **1~5 分钟**，高频任务（单测、Lint）的等待时间远超实际执行时间。
- **K8s 基础设施的优化门槛较高**：K8s 调度策略、镜像分发机制或节点预热策略等等。
- **固定资源浪费**：频繁创建/销毁 Pod 产生大量调度开销，峰值时并发受限，低谷时资源闲置。

### 1.3 为什么是现在
- 我们需要**加速反馈闭环**。
- 如果执行层不加速，未来的 AI 编码 Agent（高频率提交）会遭遇更严重的排队和延迟瓶颈。
- 需要在组织约束内，构建一个**完全由 CICD 团队自主掌控**的快速执行底座。

---

## 2. 产品目标

- **核心目标**：将高频 CI 任务（单元测试、Lint、构建）的用户感知启动时间从 **分钟级降低到秒级（< 2 秒）**。
- **架构目标**：在不依赖 K8s 任何底层优化的前提下，通过应用层“二级调度”实现任务快速分发和环境复用。采用 **声明式任务抽象 + 控制器模式**，彻底解决 P2P 推送架构的复杂性和脆弱性。
- **战略目标**：为后续“自适应编排”、“环境即函数”和“AI 自主修复闭环”提供可靠的执行层基础。

---

## 3. 功能需求

### 3.1 任务抽象 (`Task` CRD)
- **FR1.1** 系统应定义一个标准、通用的 `Task` 自定义资源（CRD），作为调度、执行和状态跟踪的唯一实体。
- **FR1.2** `Task` 的 `spec` 应包含：
    - `runtime`：指定所需的执行环境类型（如 `golang-1.22-test`）。
    - `commands`：要执行的命令列表。
    - `env`：所需环境变量。
    - `timeout`：任务超时时间。
    - `repoURL` / `commitSHA`：代码拉取信息。
    - `...`：其他执行所需的元数据。
- **FR1.3** `Task` 的 `status` 应包含：
    - `phase`：任务的生命周期阶段（`Pending` → `Scheduled` → `Running` → `Succeeded` / `Failed`）。
    - `assignedPod`：由调度器分配的，将执行此任务的 Pod 名称。
    - `message`：人类可读的状态或错误信息。
    - `startTime` / `completionTime`：执行时间戳。

### 3.2 Runtime 抽象
- **FR2.1** 系统应支持定义多种 **“Runtime”类型**。每种 Runtime 对应一个长期运行的 Deployment，Pod 内运行特定的工具链和 Agent 进程。
- **FR2.2** 每个 Runtime 类型由一组带特定 Label（如 `runtime: golang-1.22-test`）的标准 K8s Pod 组成。其副本数、资源配额等由标准 K8s 机制管理，无需自定义配置对象。

### 3.3 声明式调度器
- **FR3.1** 调度器是一个标准的 K8s 控制器，它只 **Watch `status.phase == Pending` 的 `Task` 对象**。
- **FR3.2** 对于每一个 `Pending` 任务，调度器：
    - 通过 K8s API 查询具有匹配 `spec.runtime` Label 的 Pod 列表。
    - 使用可配置策略（如最少正在执行任务数）选择其中最合适的 Pod。
    - **更新 `Task` 对象**，设置 `status.assignedPod = <选中的Pod名称>` 和 `status.phase = Scheduled`。
- **FR3.3** 调度器本身不直接与任何 Agent 通信，也不需维护 Agent 的 IP 地址或连接池。它只负责将任务绑定到 Pod。

### 3.4 主动拉取执行器 (Agent)
- **FR4.1** Agent 是每个热 Pod 内的进程，它 **Watch `status.assignedPod == 本Pod名称` 且 `status.phase == Scheduled` 的 `Task` 对象**。
- **FR4.2** 当发现新任务，Agent 应原子化地将 `status.phase` 更新为 `Running`，防止重复执行。
- **FR4.3** 随后，Agent 拉取代码，在独立的临时目录执行命令。执行期间的日志应能通过 Agent 的标准输出或引入 K8s 日志架构轻松查看。
- **FR4.4** 执行完毕后，Agent 负责更新 `Task` 对象的 `status`，将 `phase` 设为 `Succeeded` 或 `Failed`，并填充 `message` 等字段，最后彻底清理临时文件。

### 3.5 与现有 CI 系统集成
- **FR5.1** 提供轻量级 CLI 工具 `task-cli`，开发者或 Pipeline 脚本可通过 `task-cli run --runtime golang-1.22 -- go test ./...` 创建 `Task` 并轮询其状态。

### 3.6 观察与运维
- **FR6.1** 所有系统组件（调度器、Agent）都应将关键操作和错误记录日志。
- **FR6.2** 系统应暴露 Prometheus 指标：基于 `Task` 状态可以轻松统计排队时间、执行时间、失败率等。同时，平台团队应能通过 `kubectl get task` 直观地看到所有任务的实时状态。

## 4. 非功能需求

- **NFR1 隔离性**：同一 Pod 内复用任务应使用非 root 用户运行，仅可写临时工作目录。根文件系统应设为只读（`readOnlyRootFilesystem: true`），防止对 Agent 环境的意外或恶意修改。
- **NFR2 可扩展性**：调度器是一个无状态控制器，Agent 是纯粹的执行单元。所有状态都存储在 `Task` 对象中，系统水平扩展能力完全取决于 K8s 自身的性能。
- **NFR3 兼容性**：完全基于标准 K8s API 和 CRD 机制，可在任何标准 K8s 集群运行。不依赖 KEDA、Knative 等任何重量级第三方平台组件。
- **NFR4 可靠性与故障转移**：
    - **Agent 故障**：若 Agent Pod 崩溃，K8s 会重建 Pod。被分配给旧 Pod 且仍在 `Running` 的任务，将由一个全局超时检查器标记为 `Pending`，由调度器重新分配。此机制独立于 Agent 本身的稳定性。
    - **调度器故障**：若调度器有多个副本，使用 Leader Election 实现高可用。故障期间，`Pending` 任务将暂时等待，直至新的 Leader 选出，不会影响已在执行的任务。
- **NFR5 性能**：用户感知的任务启动时间（从 `Task` 创建到进入 `Running`）P95 应小于 2 秒。调度器单次处理延迟应小于 100ms。

## 5. 架构概览

```
CI 系统 (Jenkins/GA) 或 Developer
        │
        │ task-cli run --runtime golang-1.22 -- go test ./...
        ▼
   ┌─────────────────────────┐
   │  创建 Run CRD 对象      │  <─── 唯一入口
   │  (phase: Pending)       │
   └───────────┬─────────────┘
               │ Watch
               ▼
┌──────────────────────────────┐
│  调度器 (Deployment)          │
│  - 查询匹配 Runtime 的 Pod    │
│  - 设置 assignedPod & Scheduled │
└───────────┬──────────────────┘
            │ Update Task
            ▼
┌───────────────────────────────────────┐
│  Run (phase: Scheduled, assignedPod: golang-pod-a)  │
└───────────┬───────────────────────────┘
            │ Watch (by golang-pod-a)
            ▼
┌───────────────────────────────┐
│  Agent (在 golang-pod-a 内)    │
│  - 更新 phase = Running       │
│  - 执行命令 & 流式输出日志     │
│  - 更新 phase = Succeeded/Failed │
│  - 清理环境                   │
└───────────────────────────────┘
```

- **状态中心**：`Task` CRD 是整个系统的枢纽，是调度、执行、查询的唯一信源。
- **解耦点**：调度器与 Agent 完全解耦，它们之间只能通过 `Task` 对象的状态变更来感知对方的存在。
- **K8s** 仅负责管理 `Task` 对象的持久化和 `Runtime Pod` 的生命周期。

## 6. 成功指标

| 指标 | 目标值 | 测量方式 |
| :--- | :--- | :--- |
| **任务总执行时间（高频任务）** | 较之前减少 80% 以上 | 对比 `Task` 创建到完成的完整时间 |
| **用户感知等待时间 (P50)** | < 2 秒 | `Task` 创建到 `Running` 的时间 |
| **用户感知等待时间 (P95)** | < 5 秒 | 同上 |
| **系统故障率** | < 0.5% | 非用户代码失败的任务数 / 总任务数 |
| **池资源利用率** | > 60% | (忙碌 Pod·时间) / (总 Pod·时间) |

## 7. 风险与缓解

| 风险 | 缓解措施 |
| :--- | :--- |
| **Pod 内多任务隔离不足** | 首先只对可信内部任务开放；使用非 root、只读根文件系统、`emptyDir` 隔离；后续可选引入嵌套 gVisor 沙箱 |
| **池耗尽导致排队过长** | `Task` 的`排队时间`指标可直观呈现；提供手动/自动扩容 Runtime Deployment 的能力 |
| **Agent 内存泄漏/状态污染** | 设置 Agent 最大存活任务数，执行完 N 个任务后自我终止，由 K8s 重建干净 Pod |
| **CRD 和控制器开发复杂度** | 使用成熟的 Operator 框架（如 Kubebuilder）来生成脚手架和客户端库，专注于业务逻辑 |

---

## 8. 未来规划

- **v2.1**：引入一个 **超时/僵死任务回收器**，自动将长时间 `Running` 但 Agent 已消失的任务重设为 `Pending`。
- **v2.2**：与 Phase 1 诊断机器人联动，当 `Task` 失败时，自动触发根因分析和修复 PR 创建。
- **v3**：为 `Task` 增加 `sandbox` 选项，支持按需启动强隔离沙箱（Firecracker/gVisor），实现更彻底的“环境即函数”。
- **v4**：支持声明式意图（如“验证 API 兼容性”），AI 编排器自动生成并创建一系列 `Task` 对象来满足该意图。
